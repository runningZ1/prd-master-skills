# PRD 实战指南 - 如何撰写清晰有效的PRD

**来源**: pagerank.ing - 产品先生

---

## 一、PRD 常见问题识别

以下是一个"注册页"PRD的常见问题示例：

### 1. 重复性信息
电子邮件和密码的提示标题重复，会导致阅读者混淆，工程师会来询问是否写错。

### 2. 本地化与国际化
PRD是否考虑到多语系支持？从提示文字来看，目前似乎只有繁体中文版本。

### 3. 用户场景定义
从PRD无法得知：
- 这个界面是在APP上还是网页上？
- 如果是网页，是否只针对桌面设计？
- 还是自适应网页（RWD - Responsive Web Design）？

**重要提醒**：你自己找不到的问题，一定要靠团队帮你找出来，因为实际要做的是他们。

## 二、PRD 的六大核心架构

### 1. 修订追踪表

为了保持PRD文件的清晰和准确性，对文件的每次修改都需要记录。

**作用**：
- 向团队展示更改发生的具体位置
- 避免团队每一页都要看到底改了哪里
- 对回顾和自我检讨非常有帮助

**表格格式**：

| 版本号 | 修改日期 | 修改内容 | 修改人 | 影响范围 |
|--------|----------|----------|--------|----------|
| V1.0 | YYYY-MM-DD | 初始版本 | XXX | 全部 |
| V1.1 | YYYY-MM-DD | 修改登录流程 | XXX | 第3章 |

### 2. 背景简介

**这一段非常关键**（但大部分工程师不会看）。

它要说清楚：
- 产品是怎么被提起来的
- 背后的市场问题是什么
- 跟公司整体业务有什么关联

#### 三个核心要素：

**目标**：给大家一个清晰、热血的"远方"（愿景）！
- 我们拼死拼活做这个，最终是想达成什么？
- 让团队知道瞄准哪里开火

**背景**：讲"起心动念"的故事
- 为什么会有这个想法？
- 是看到市场哪里在唉唉叫？
- 还是我们自己哪里卡卡非解决不可？

**价值**：让大家眼睛一亮！
- 这产品在市场上"是要打哪个山头"（定位）？
- 凭什么我们做得出来、还能赢？
- 秀肌肉、秀机会，证明这东西有搞头、有市场！

**记住**：我们不是为了写文件而写，而是为了说服一群人：这个产品值得做，做出来真的有用。

### 3. 功能清单

功能清单就是整份PRD最硬、最不能打马虎眼的部分！

**为什么重要**：
- 白纸黑字列清楚：这个产品到底要做哪些功能？具备什么特性？
- 整个团队的共同作战地图
- 避免"我以为..."的灾难
- 开发的指北针，告诉团队火力要集中在哪
- 验收的生死状，产品做完是不是及格？

**撰写原则**：
- 具体、明确，一条一条列出来
- 像开菜单一样清清楚楚
- 不要模糊描述或天花乱墜

### 4. 功能流程

**核心理念**：文字描述流程讲到烧声，还不如直接上图！

#### 为什么流程图这么重要？

**最强的翻译蒟蒻**：
- 管你是复杂的工作流程还是使用者怎么点来点去
- 图一画下去，逻辑、架构、谁接谁的棒，三秒钟全部现形

**专治复杂症头**：
- 牵扯八百个单位、步骤多到靠北的流程？
- 流程图能拉直给你看，清清楚楚

**统一理解频道**：
- 工程师看架构
- 设计师看互动
- 业务看使用者动线
- 一张图搞定所有角度的理解

#### 最狠的功能：叫人心甘情愿画押！

流程图一摊开：
- 哪个步骤卡在哪个单位手上
- 谁要负责接球协作
- 白纸黑线画得明明白白

**相关单位看懂了、没话说了？**
- 好！这张图就是你们的军令状
- 代表你们都认了："这一段，你负责！"
- 后面想推拖拉说"啊我当初没看清楚"？门都没有！

**流程图的三大作用**：
1. **釐清责任的图**：谁的地盘谁顾好，图上标得清清楚楚
2. **打通协作的任督二脉**：跨部门要一起跑？图拿出来照着走
3. **产品不出包的保险**：对着图做，关键步骤和互动想漏掉都难

### 5. 功能结构

功能结构就是**把产品的功能，照不同类型"分门别类"，再用视觉化的方式画出来**。

#### 三大价值：

**全局观**：
- 整个产品由哪些大功能、小功能组成
- 彼此之间怎么串、怎么分层
- 一目了然

**防漏小帮手**：
- 照着图开发，比较不容易遗漏角落的功能或该有的页面
- 确保产品是完整的

**关系理清楚，评估有依据**：
- 一眼就看出各个功能模块之间的"逻辑关系"
- 谁是老大（核心功能）、谁是小弟（子功能）
- 谁跟谁是邻居（关联性）
- 所有团队要估工作量、排时程，全靠这张图当参考基准

### 6. 页面设计（Wireframe）

**Spotify案例**：Spotify那张看起来像随手涂鸦的Wireframe，现在可是值700亿美金！

#### Wireframe 到底用来做什么？

简单讲，它就是产品"骨架的草稿"。

**核心原则**：
- **专注"骨头"，不看"皮相"**：只管功能怎么摆、版面怎么排、使用者要怎么点来点去
- **核心是"布局"和"互动"**：按钮放哪里？表单长怎样？使用者从A点到B点怎么走？

#### 重要警告：手伸进设计师的饭碗！

有些资深设计师看到产品经理动版面，会在需求会议上直接开骂：
- "你凭什么这样排？"
- "这不符合UX原则！"

#### 生存守则：

1. **搞清楚你的角色**：
   - 你是产品经理，不是设计师！
   - 目的是"定义产品的画面基本架构"，不是在做设计！
   - 促进沟通协作

2. **别越界**：
   - 绝对不要花时间去喬颜色、字体、阴影、圆角要几pixel
   - 那不是你的战场！
   - 把架构和互动逻辑讲清楚就好

3. **保持同步，别让工程师问看谁的**：
   - 设计师后续会根据你的Wireframe产出美美的UI稿
   - **当UI稿出来后，你的Wireframe也要跟着更新到最新版本！**
   - 避免工程师看到旧版Wireframe和新版UI稿不一样而困惑

#### Wireframe 的精髓：

- **快、狠、准**：快速勾勒出核心架构和互动，别纠结细节
- **沟通基石**：让团队在"产品该长什么骨架"上达成共识
- **活的文件**：随着设计稿演进，同步更新

**记住**：Wireframe要去掉Look and feel（视觉和感觉），重点是确认产品逻辑！

## 三、如何撰写清晰有效的PRD

### 1. 清晰表达：避免模糊与冗长

PRD最常被砲什么？
- "写得落落长，看完还是雾煞煞"
- "需求像雾里看花，各自解读不同方向"

#### 一图解千言

**为什么要用图**：
- 图一摆出来，产品流程或功能逻辑马上变直球
- 团队三秒就能抓到重点
- 不用在那边猜心

**实例**：
- 今天要讲订单怎么跑
- 与其写一大段"用户下单后系统如何如何..."
- 不如直接一张流程图出来
- 从用户下单、付钱、仓库接单到出货
- 每个环节用箭头标得清清楚楚

**效果**：
- 保证团队误解的机率砍半
- 省掉你后面十次确认会议！

#### 条列式呈现

条列式清单不仅结构明确，而且能够帮助阅读者快速定位重点。

**条列式的优势**：
- **改需求不折寿**：要加新规格或砍旧逻辑？直接插入、删节点
- **层次瞬间分明**：主功能、子规则谁大谁小，一眼看穿
- **抓重点就是快**：不用拿荧光笔涂满整页，扫过去要害自动浮现

**条列式结构示例**：

- **功能名称**：搜索框功能
- **触发条件**：点击按钮、按下Enter键
- **可输入资料**：文字、数字、符号
- **功能范围**：前端UI、后端资料库及搜索机制
- **处理逻辑**：输入验证、关键字传递、结果打包
- **错误处理**：空白输入提示、后端超时提示

#### 不要连续性的文字呈现

**反例**（连续文字说明）：
```
这个功能是"搜索框功能"，当使用者点击搜寻按钮或直接按下Enter键时就会触发，可以支援输入文字、数字以及符号。此功能涵盖了前端的UI呈现，以及后端资料库的资料检索与搜索机制...
```

这种连续文字结构让人读起来较为困难。

#### 最好能面对面确认

建议在文档撰写完成后，进行1对1的面对面确认：
- 当场解答疑问
- 根据研发、测试、设计等部门的反馈实时调整
- 面对面沟通比纯文字交流更高效
- 避免因理解偏差而导致的后续问题

### 2. 分好项目类型：既有项目与新增需求的划分

产品经理写PRD还有一个致命伤：老是把"旧功能改版"和"全新功能"混在同一锅大杂烩里！

**问题**：
- 团队根本分不清哪些是旧东西微整形
- 哪些是从零长出来的新需求
- 最后不是漏改就是多做白工

**解决方案**：

#### 分页显示
- 直接开独立区块，甚至分不同页面
- 专门写"旧功能"和"新功能"
- 让读者在阅读时有更清晰的脉络

#### 提供历史连结
- **每条需求都要超连结连回旧版定义**
- 让团队点一下就能穿越时空看前版长怎样
- 省掉在那边考古找文件的时间
- 这招一用，保证需求回溯不迷路

**好处**：
- 便于追溯历史修改记录
- 帮助新加入团队的成员迅速了解需求演变过程
- 节省时间，让团队对产品发展脉络有更深入认识

### 3. 数据与流程：资料来源与系统串接的详细描述

对资料来源及系统串接需求的描述是重中之重。

#### 明确标注资料流向

对于任何涉及数据处理的功能，必须在PRD中标明：
- 数据的流向
- 存储位置（前端或后端）

**完整流程**：
- 数据从用户端的输入
- 到后端的处理
- 再到最终显示给用户

**实例**：
- 描述搜索功能时
- 详细说明资料从前端输入到后端搜索引擎
- 再回传结果的每一个步骤

**好处**：
- 降低错误的发生率
- 方便日后进行效能优化与错误排查

#### 系统串接需求

在当今复杂的系统环境中，产品往往需要与第三方系统进行串接。

**以第三方发票系统为例**：
- PRD中除了需列明参与串接的人员
- 还必须详细描述串接后的流程
- 包含正向与逆向流程图

**流程图要求**：
- 标示清楚每个环节的责任范围
- 避免模糊空间
- 研发团队能够一目了然地看到数据流动与功能切割
- 更好地进行开发与测试

### 4. 使用表格呈现复杂的内容

当需求变得复杂，单靠文字往往难以表达清楚所有细节。

#### 表格应包含哪些栏位？

针对例如"搜寻框"这类功能，表格中可以包含：

| 栏位 | 说明 |
|------|------|
| **功能名称** | 清楚说明功能本身的名称 |
| **触发条件** | 详细列出触发该功能的行为（如按钮点击、键盘操作） |
| **可输入资料** | 列举允许的数据类型与格式 |
| **功能范围** | 区分前端与后端的责任与处理范围 |
| **处理逻辑** | 步骤式描述功能的实际运作流程 |
| **错误处理** | 对于常见错误状况，提供相应的提示与解决方案 |
| **测试案例** | 包含正向与逆向测试的具体案例 |
| **效能标准** | 明确功能在效能上的要求（响应速度、并发数量） |

#### 表格的灵活运用

- 并非所有情况都必须使用表格
- 对于内容较为简单或直观的部分，可以直接使用条列或段落描述
- 对于需要精确量化和分层次展示的需求，表格无疑是最佳工具

**建议**：
- 先草拟出一个简单的框架
- 根据不同功能或模组的复杂度
- 决定是否使用表格进行详细说明

### 5. 务必考虑Corner Case

在PRD中，除了主流程外，各种异常情况（Corner Case）的考量也至关重要。

#### 正常与异常流程并重

撰写PRD时，不能仅考虑理想状态下的正向流程，必须全面覆盖所有可能出现的情况。

**例如**：
- 当用户输入不符合规定的字元或超过限定长度时，系统应如何处理？
- 在系统忙碌、后端无法及时回应时，前端应提供何种提示？

**重要性**：
- 这些细节在设计测试案例时尤为重要
- 能够确保产品在面对各种突发状况时依然稳定运行

#### 灵感来自生活

**建议**：
- 回家散步边走边想
- 在轻松的环境中，大脑反而能够跳脱固有思维模式
- 发现那些在办公室中未曾注意到的逻辑漏洞与潜在问题
- 利用碎片时间进行思考与总结
- 从不同角度审视需求的全面性

### 6. 解决专有名词疑惑与全局规划

在PRD撰写中，常常会遇到一些专有名词或术语：
- LOG纪录
- HK机制
- Retry机制
- 前置条件
- 系统关联

#### 解决方法

**增加术语解释页**：
- 在PRD中专门增加一页
- 用以解释与定义这些术语
- 确保所有团队成员对关键词有统一的认知

**参考最完整的范例**：
- 在撰写正式文档前，先"看公司最完整的来参考"
- 询问研发团队他们的阅读与理解方式
- 有助于快速上手
- 在学习中进行有效改进
- 逐步形成适合自己团队的PRD撰写风格

## 五、关键提醒

### 1. PRD 的延续性

正常的公司，规格通常是会有"延续性"的：
- 每一个阶段都会纪录下来
- 留给"后人"去做产品规格的瞭解
- 而不该是让人"考古"公司的产品应该是什么样的

### 2. 没有 PRD 的后果

如果今天你想知道某个系统"到底藏了什么功能"，但公司告诉你"想看？得先关掉服务让大家没得用才行"，你会这样干吗？

**没PRD的问题**：
- 新人或团队想看功能规格时
- 难道只能"把系统停机，一页一页，把画面翻出来考古"？
- 还是要跪求当初的工程师通灵回忆？

**新人的绝望**：
- 进公司第一天
- 面对一座没地图（PRD）的产品迷宫
- 是要擲筊问神明"这个按钮为什么放这里"？
- 还是要烧香拜托离职工程师託梦？

### 3. 团队协作的重要性

**你自己找不到的问题，一定要靠团队帮你找出来，因为实际要做的是他们。**

建议在产品需求说明会议之前：
- 先找其他单位的人沟通
- 以避免在正式会议上变成批斗大会

## 六、总结

撰写一份清晰、结构合理且易于理解的PRD并非易事，但它对整个产品开发流程有着不可估量的影响。

**核心要点**：
- 从语句表达、内容结构
- 到资料来源、流程细节
- 再到异常情况与专业术语的解释
- 每一个环节都需要细心斟酌

**持续改进**：
- 不论你是初入职场的新手产品经理
- 还是经验丰富的老鸟
- 持续反思与改进都是提升自己能力的关键
- 每一次的修改与反馈，都是一次成长的机会

**行动建议**：
- 从每一个细节做起
- 打造一份真正能够促进团队合作的PRD
- 减少误解并提升产品质量
